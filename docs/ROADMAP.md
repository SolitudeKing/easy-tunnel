# EasyTunnel 产品与版本路线图

## 1. 当前产品基线

EasyTunnel 当前已经解决最核心的问题：把一条 OpenSSH 本地转发命令变成可保存、可观察的一键开关。

| 能力 | 当前状态 | 主要限制 |
| --- | --- | --- |
| SSH 本地转发 | 已实现单会话多条 `-L` | 不支持反向转发、SOCKS |
| 身份认证 | 私钥路径、BatchMode | 私钥必填，缺少 ssh-agent 状态引导和口令方案 |
| 隧道状态 | 连接中、已连接、停止中、错误 | 没有重连、降级、目标服务状态 |
| 服务快捷打开 | RDP、Web、TCP | 没有自定义协议、启动参数和服务模板 |
| 配置管理 | JSON 原子保存、schema v1→v2 迁移、SSH 命令导入 | 没有备份恢复和通用配置包导入导出 |
| 安全默认值 | 参数列表、禁用 shell、仅回环绑定 | 首次主机密钥确认不可视、没有应用独立 known_hosts |
| 日志 | 本次运行内存日志 | 重启即丢失、不能导出、没有诊断包 |
| 应用生命周期 | Page 断开时清理进程 | 没有系统托盘、单实例和强制退出兜底 |
| 发布 | 源码、Windows 安装包、GitHub Release 自动更新 | 尚未签名，缺少多平台桌面产物 |

## 2. 产品目标

未来几个版本应围绕四个目标推进：

1. **可靠**：用户能够相信开关状态；断线会被发现，退出不会遗留未知 SSH 进程。
2. **可诊断**：连接失败时给出原因、修复建议和可导出的上下文。
3. **高效率**：已有 SSH 命令可以导入；常用隧道可以复制、分组、自动启动。
4. **可扩展**：在不破坏安全默认值的前提下支持多跳、反向转发和 SOCKS。

非目标：EasyTunnel 不应演变为完整 SSH 终端、远程文件管理器或私钥编辑器。这些功能会显著扩大安全边界，也偏离“隧道管理开关”的核心定位。

## 4. 分阶段版本建议

### 4.1 `0.2.0`：可靠运行

目标：先让当前的本地转发在长期运行和网络波动下可信。

#### A. 进程生命周期可靠化（P0）

交付内容：

- Windows 使用 Job Object 或等价进程监管，使应用退出时自动回收 `ssh.exe` 及可能的代理子进程。
- 增加应用单实例锁，第二次启动时激活已有窗口，而不是重复启动自动连接。
- 显式处理“关闭窗口”：询问“断开并退出 / 最小化到托盘 / 取消”。
- 记录本应用启动的 PID、创建时间、命令摘要；下次启动只在多重校验后提示清理孤儿进程，不能只按 PID 杀进程。

验收标准：

- 连接 1 条和 10 条隧道后正常关窗，所有本次创建的 SSH 进程在 3 秒内结束。
- 启动与退出同时发生、连接中退出、停止超时等竞态都有自动化测试。
- 连续双击启动只存在一个 EasyTunnel 主实例。

#### B. 自动重连（P0）

交付内容：

- 每条隧道独立设置是否重连、最大次数、初始间隔和最大间隔。
- 使用指数退避并加入少量随机抖动，避免服务器恢复时所有隧道同时连接。
- 用户主动断开后绝不自动重连；认证失败、主机密钥变化等永久错误默认不重试。
- UI 增加“等待重连”和“第 N 次重试”状态，并允许立即重试或取消。

验收标准：

- 模拟 SSH 异常退出后按策略重连；用户关闭开关后重连任务立即取消。
- 网络不可用时不产生忙循环，不在短时间内创建大量 SSH 进程。
- 每次尝试都有结构化原因、序号和下一次时间。

#### C. 双层健康状态（P0）

当前本地端口监听只能证明 SSH 转发建立，不能证明远端服务可用。新版本应分别展示：

- **隧道状态**：SSH 进程和本地监听是否正常。
- **服务状态**：通过隧道连接目标端口是否成功、耗时多少。

服务探测必须可关闭并限制频率。某些服务会记录每次 TCP 连接，因此默认只在连接建立后探测一次，周期探测由用户显式开启。

验收标准：SSH 隧道正常但目标 RDP 关闭时，卡片显示“隧道已连接 / 目标不可达”，而不是笼统的连接失败。

#### D. 真实设置页（P0）

至少支持：

- SSH 可执行文件路径与“检测”按钮。
- 默认私钥目录。
- 默认连接超时、保活和重连策略。
- 日志级别、保留天数和最大磁盘空间。
- 关闭窗口行为、开机启动、主题模式。

设置保存失败不能改变运行时状态；不支持的平台选项应隐藏或明确禁用。

#### E. 配置恢复（P0）

- 每次成功保存前保留最近 3～5 个版本的备份。
- 配置损坏时进入恢复页，展示错误和备份列表，不直接用示例配置覆盖原文件。
- 提供“导出脱敏诊断信息”和“打开配置目录”。

#### F. 一次点击“连接并打开”（P0）

当前未连接时需要先打开开关，再等待并点击“打开服务”。卡片主操作应变为：

- 未连接：`连接并打开远程桌面/网页`。
- 连接中：显示进度并允许取消。
- 已连接：`打开远程桌面/网页`。
- 失败：`重试`，同时保留诊断入口。

“打开”是一次性用户意图，只能在本次 generation 成功后执行一次。自动连接、自动重连不能意外重复弹出 RDP 或浏览器。仍保留“仅连接”作为次级操作。

验收标准：从未连接状态点击一次，成功时只打开一个客户端；快速连续点击、失败后迟到成功、重连恢复均不会重复打开。

#### G. 表单和连接诊断（P0/P1）

- 常用字段与高级 SSH 参数分层，暴露当前模型已有但 UI 尚未编辑的超时和保活。
- 字段旁显示校验错误，端口冲突可推荐空闲端口。
- 提供分步骤“测试连接”：OpenSSH、私钥/agent、本地端口、SSH 认证、转发和目标服务。
- 表单响应窗口尺寸，支持 Tab、Enter、Esc 和未保存修改确认。

### 4.2 `0.3.0`：高效管理

目标：让已有命令和多条隧道的管理更顺畅。

#### A. 从命令导入（基础能力已完成）

已支持粘贴类似以下命令并解析为表单：

```text
ssh -i .\pi-server -p 22 -L 13389:192.168.3.88:3389 pi@pi.example.com -N
```

只解析受支持的参数，绝不执行输入文本；`ProxyCommand`、`LocalCommand` 等高风险参数会被拒绝。后续增强项是更细粒度地展示未知参数位置，以及提供可选择的修正建议。

#### B. 批量管理（P1）

- 复制隧道、标签、分组、收藏和拖拽排序。
- 分组连接/断开，但操作需显示逐条结果，不能用一个总开关掩盖部分失败。
- 支持按名称、主机、目标、标签和状态过滤。
- 提供配置导入导出；私钥文件和口令不进入导出包。

#### C. 托盘增强与通知（P1）

- `0.2.0` 先完成基础驻留和安全退出；本阶段再让托盘显示“已连接数 / 错误数”，并提供常用隧道快捷开关。
- 断线、重连成功和永久失败可以通知；相同错误需要合并，避免通知风暴。
- 托盘退出必须走统一 shutdown 流程。

#### D. 诊断中心（P1）

- 汇总 OpenSSH 版本、配置路径、端口占用、密钥可读性、ssh-agent 状态和最近错误。
- 一键生成脱敏 ZIP：应用版本、平台信息、结构化日志和去敏配置，不包含私钥、用户名可选脱敏。
- 每个错误给出“原因 / 建议操作 / 原始信息”三层内容。

### 4.3 `0.4.0`：高级 SSH 场景

目标：扩展连接能力，但保持默认界面简单。

- ProxyJump 多跳连接，跳板机作为可复用实体管理。
- `-R` 反向转发和 `-D` SOCKS；用明显的方向图和安全警告区分。
- 一个 SSH 会话承载多条转发，将“SSH 会话”和“端口映射”拆为两层。
- 支持系统 SSH config 的可控模式：默认仍使用界面参数；高级用户可选择指定 config 文件，并在预览中显示其影响不可完全展开。
- ssh-agent 状态检测、密钥指纹显示和加密私钥引导。

验收底线：任何新转发模式仍必须使用参数数组、禁止 shell、校验监听范围，并在命令预览中准确呈现。

### 4.4 `1.0.0`：可交付桌面产品

- Windows 安装包、应用图标、版本资源、卸载流程和代码签名。
- 自动化构建、测试、依赖审计和产物校验和。
- 配置迁移向后兼容，支持从至少两个旧版本升级。
- 崩溃恢复、日志轮转、无障碍键盘操作、125%～200% 缩放测试。
- 更新检查和明确的回滚路径；自动更新本身可在后续版本实现。

## 5. 推荐实施顺序

依赖关系如下：

```text
结构化事件与新状态机
├─ 自动重连
├─ 服务健康检查
├─ 托盘状态与通知
└─ 持久日志与诊断中心

AppSettings + 配置迁移
├─ 真实设置页
├─ 日志策略
├─ 关闭行为
└─ 开机启动

Session / Forward 模型拆分
├─ 多条转发
├─ ProxyJump
├─ 反向转发
└─ SOCKS
```

建议按六个短迭代执行：

1. 事件模型、状态机、进程监管和关闭竞态。
2. 重连策略、取消语义和失败分类。
3. 目标服务健康检查、卡片双层状态。
4. AppSettings、配置备份恢复、真实设置页。
5. 持久日志、诊断导出、错误操作建议。
6. 复制/分组和系统托盘，并继续改善已经落地的命令导入体验。

每个迭代结束都应保持主分支可运行，不建议一次性重写整个 Flet 界面。

### 5.1 两种范围选择

默认推荐上述“可靠性优先”顺序。如果下一阶段的目标是尽快给新用户演示或验证需求，可采用“上手体验优先”的精简范围：

1. 一次点击“连接并打开”。
2. SSH 命令粘贴导入。
3. 表单分层、字段级错误和空闲端口推荐。
4. 最小自动重连。
5. 进程关闭竞态和配置备份仍作为不可删除的安全底线。

这种范围能更快改善首次使用，但不能用它取代 Job Object、恢复模式和状态机等可靠性基础。高级转发仍应后置。

## 6. 成功指标

指标应默认只在本地计算，不上传遥测：

- 隧道启动成功率和启动耗时分位数。
- 异常断线后成功恢复比例及平均恢复时间。
- SSH 进程残留数，目标必须为 0。
- 用户看到“未知错误”的比例。
- 配置恢复成功率。
- 安装后首次成功建立隧道所需步骤数。
- 从保存的未连接隧道到打开目标服务的主要点击数，目标为 1。

如未来引入遥测，必须默认关闭、明确告知采集字段，并且不得采集主机、用户名、内网地址、私钥路径或原始命令。

## 7. 暂不建议优先实现

- 云端账户和配置同步：需要新的信任、加密和隐私模型。
- 内置 SSH 密码/私钥口令保存：应先接入系统凭据库并完成威胁建模。
- 自定义任意 Shell 启动命令：会破坏当前无 Shell 的安全边界。
- 完整终端和 SFTP：与隧道管理目标差异过大。
- 插件市场：在核心模型和版本兼容策略稳定前维护成本过高。
