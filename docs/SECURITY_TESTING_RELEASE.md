# 安全、测试与发布指南

## 1. 适用范围

EasyTunnel 会启动系统 OpenSSH、读取用户选择的私钥路径、监听本地端口并访问远端网络，因此即使界面简单，也属于安全敏感的桌面网络工具。本指南覆盖：

- 输入与命令执行边界。
- 主机身份、私钥、配置和日志保护。
- 子进程与监听端口生命周期。
- 自动化测试和 Windows 发布准入。

当前安全基线值得保留：使用参数数组和 `shell=False`、禁用交互密码认证和本地命令、默认回环绑定、私钥只保存路径、JSON 原子替换。

## 2. 信任边界与资产

### 2.1 需要保护的资产

- 私钥文件及其路径、用户目录结构。
- SSH 用户名、服务器地址、内网 IP 和端口拓扑。
- known_hosts 中的主机身份信息。
- 当前活动转发和本地监听端口。
- 配置、日志、诊断包和剪贴板内容。
- 应用选择并执行的 `ssh.exe` 与本地客户端。

### 2.2 不可信输入

- 用户粘贴的 SSH 命令和手填字段。
- 导入的配置文件。
- OpenSSH stdout/stderr。
- PATH 和自定义可执行文件路径。
- 远端主机密钥、DNS 结果和网络响应。
- 升级包、依赖和构建环境。

## 3. 威胁与控制

| 风险 | 现状 | 下一阶段控制 | 验收方法 |
| --- | --- | --- | --- |
| Shell/参数注入 | 已使用参数数组、`shell=False` | 所有新功能继续使用结构化字段；命令导入只解析不执行 | 元字符、换行、前导 `-`、引号、Unicode 模糊字符负向测试 |
| PATH 劫持 `ssh.exe` | 当前优先 `shutil.which` | Windows 默认优先 `%WINDIR%\System32\OpenSSH\ssh.exe`；自定义路径需用户明确选择 | PATH 首位放置假 `ssh.exe`，默认路径不得执行它 |
| 首次连接中间人 | `accept-new` 是 TOFU | 应用独立 known_hosts；首次显示指纹；支持管理员预置 SHA256 指纹 | 新主机、已知主机、指纹不匹配、密钥变更测试 |
| 监听暴露到局域网 | UI/核心限制回环 | 继续默认回环；若以后允许远程客户端，必须单独授权并显示风险 | `0.0.0.0`、`::`、LAN IP 导入及 UI 绕过测试 |
| 私钥泄漏 | 只保存路径 | 不复制私钥；检测 ACL/网络共享；口令只交给 ssh-agent/系统凭据库 | 配置、日志、诊断包和发布产物秘密扫描 |
| 剪贴板泄漏拓扑 | 可复制完整命令 | 默认复制脱敏命令；复制完整命令前提示剪贴板历史风险 | 剪贴板文本检查及超时清除行为测试 |
| 应用退出后进程残留 | 正常 shutdown，强杀仍有风险 | Windows Job Object、统一关闭截止时间、单实例 | 正常退出、崩溃、任务管理器强杀后检查进程 |
| 多实例覆盖配置 | 仅进程内写锁 | 命名互斥量、文件 revision、保存冲突提示 | 双实例并发连接和保存测试 |
| 日志注入/污染 | 原始 SSH 输出直接展示 | 结构化事件；控制字符净化；日志字段长度上限；脱敏 | ANSI/换行/超长/非法编码 stderr 测试 |
| 供应链污染 | 源码运行时 pip 安装 | 正式包自包含、锁文件、SBOM、签名、CI 构建 | 无 Python/无网络 VM 安装；签名和哈希验证 |

## 4. SSH 可执行文件可信策略

### 4.1 默认选择顺序

Windows 建议：

1. 明确配置且用户确认的自定义绝对路径。
2. `%WINDIR%\System32\OpenSSH\ssh.exe`。
3. PATH 中结果只作为诊断候选，不应在无提示时优先执行。

设置页显示：绝对路径、`ssh -V` 结果、文件版本、发布者/签名状态（能检测时）。路径变化需要重新确认。不能自动从网络下载替换系统 SSH。

macOS/Linux 可使用可信系统目录与 PATH，但仍记录最终绝对路径并拒绝目录或非普通文件。

### 4.2 SSH config 模式

当前 `-F NUL`/`/dev/null` 确保实际参数主要来自界面，这是清晰的安全默认值。以后若支持用户 SSH config，应提供显式模式：

- 用户选择具体 config 文件。
- UI 告知 ProxyCommand、Match、LocalForward 等配置可能改变行为。
- 默认拒绝或警示 `LocalCommand`、`PermitLocalCommand` 等执行能力。
- 诊断页显示生效模式，但不要把可能含秘密的整个配置复制进日志。

## 5. 主机密钥和 known_hosts

### 5.1 推荐流程

1. 每个应用实例使用 `%APPDATA%\EasyTunnel\known_hosts`，不静默修改用户全局文件。
2. 新主机连接前获取候选公钥并计算 OpenSSH SHA256 指纹。
3. 显示主机、端口、密钥算法和指纹，要求用户确认或与管理员提供值比较。
4. 保存确认记录后使用 `StrictHostKeyChecking=yes` 建立正式连接。
5. 密钥变化硬失败，UI 不提供“一键忽略并连接”。

注意：`ssh-keyscan` 能获取网络对端提供的公钥，但不能独立证明对端身份。必须提示用户通过可信渠道核对指纹；不能把“扫描到”描述为“已验证”。

### 5.2 验收场景

- 标准 22 端口和自定义端口的 known_hosts 格式。
- Ed25519、ECDSA、RSA 密钥。
- 同一主机多个算法、主机名和 IP 地址差异。
- 首次拒绝、首次确认、已知密钥、密钥轮换和恶意替换。
- 并发首次连接不能写坏 known_hosts。

## 6. 私钥与身份认证

- 私钥路径必须是绝对路径、普通文件；不读取或解析私钥内容用于日志。
- Windows 诊断应检查 ACL 是否明显过宽、是否位于网络共享、OpenSSH 是否能读取。
- 加密私钥优先通过 Windows OpenSSH Authentication Agent；应用只检测和引导，不保存口令。
- 以后若加入 Credential Manager/DPAPI，只保存确有必要的短期凭据；私钥文件本身仍不进入凭据库。
- `IdentityAgent`、Pageant 等扩展应作为独立认证模式建模。
- 导出配置只包含一个“需要重新选择私钥”的引用选项，默认不泄漏完整本地路径。

## 7. 配置、日志与诊断数据

### 7.1 文件权限

Windows 创建 `%APPDATA%\EasyTunnel` 时，验证 ACL 不允许普通其他用户修改；保留当前用户、SYSTEM 和管理员所需权限。若无法收紧，不应阻止启动，但需在诊断中心显示警告。

配置备份同样受保护。临时文件必须创建在同一受保护目录，成功替换或失败后清理。

### 7.2 日志脱敏规则

建议集中实现 `Redactor`，至少处理：

- 私钥绝对路径 → `<KEY_PATH>/filename`。
- 用户名 → 可配置为 `<USER>`。
- SSH 主机、内网目标 → 诊断包中替换或稳定哈希。
- 命令参数 → 保留选项名和端口语义，隐藏本地身份信息。
- 控制字符和 ANSI escape → 移除或可见转义。
- 单条输出长度 → 设置上限并标记截断。

禁止记录：私钥内容、口令、完整环境变量、Windows 凭据、ssh-agent 密钥材料。

### 7.3 默认示例数据

当前源码示例包含具体域名、内网 IP 和私钥文件名，适合本项目内部演示，但正式公开发布前应：

- 改用 `example.com` 等保留域名和文档地址段，或使用空白引导页。
- 取消自动扫描项目上级目录中的特定私钥。
- 检查 wheel、安装包、README、测试夹具和截图中没有真实连接信息。

## 8. 自动化测试策略

### 8.1 测试分层

| 层级 | 目标 | 典型工具/替身 | 建议准入 |
| --- | --- | --- | --- |
| Model 单元测试 | 校验、状态转移、错误分类、迁移 | 纯 Python、FakeClock | 核心分支覆盖率 ≥ 85% |
| Repository 测试 | schema 迁移、原子保存、更新校验 | 临时目录、假 HTTP Response | 失败不破坏已有数据 |
| Service 测试 | SSH argv、进程、端口探测、平台适配 | FakeProcess、FakePortProbe | 平台参数和停止路径正确 |
| ViewModel 用例测试 | start/stop/retry/health/save 编排 | Fake Service、临时 Repository | 所有状态和取消路径通过 |
| 子进程契约测试 | argv、stdin/out、隐藏窗口、kill fallback | 可脚本化假 ssh 可执行文件 | Windows/Linux 平台参数正确 |
| 真实 OpenSSH 集成 | 认证、host key、转发、断网 | 隔离 sshd 容器或测试 VM | 不依赖公网或生产服务器 |
| Flet UI 测试 | 主视图、表单、状态和错误交互 | 构造测试 + 桌面 E2E | 固定 Flet 版本全部通过 |
| 打包 E2E | 安装、启动、连接、升级、卸载 | 干净 Windows VM | 标准用户、无 Python、无网络可运行 |
| 稳定性测试 | 长时间、批量、句柄泄漏 | 20 条假隧道、24h soak | 内存/线程/句柄不持续增长 |

### 8.2 必测并发场景

- 快速开→关→开，最终只存在一条与最后意图一致的连接。
- `Popen` 返回前关闭应用。
- 旧 generation 的 stdout/退出回调晚于新连接到达。
- 重连计时器触发同时用户手动停止。
- 删除/编辑配置同时监控线程退出。
- 10～20 条隧道自动连接，受并发上限控制。
- 多个进程忽略 terminate，shutdown 在全局截止时间内 kill。
- 日志第 301、1000 和 10000 条仍能更新 UI 且内存有界。

### 8.3 真实 OpenSSH 场景

- Ed25519/RSA；正确密钥、错误密钥、加密密钥 + agent。
- 新主机、已知主机、主机密钥变化。
- DNS 失败、拒绝、超时、连接中断、保活失败。
- IPv4/IPv6、本地端口抢占、目标服务关闭。
- 睡眠/唤醒、VPN/Wi-Fi 切换。

测试服务器必须隔离且使用一次性密钥，不能依赖示例中的真实服务器。

### 8.4 UI 兼容性回归

项目已出现过 Flet 0.28.3 API 差异，因此每次依赖或界面改动必须执行：

1. 三个主视图和编辑弹窗的控件构造测试。
2. `mount()` 页面会话创建测试。
3. 文件选择器、对话框、SnackBar、Dropdown 的交互测试。
4. 100%、125%、150%、200% DPI 下的截图/布局基线。
5. 键盘 Tab/Enter/Esc 流程和屏幕阅读标签检查。

完整传递依赖应生成锁文件；升级 Flet 必须由单独 PR 完成，不能和业务功能混在一起。

## 9. CI 流水线建议

每个提交：

```text
format/lint → type check → unit tests → integration tests
            → secret scan → dependency audit → build smoke
```

建议门禁：

- Ruff 格式和 lint。
- Pyright 或 Mypy；Model、Repository、Service、ViewModel 先达到 strict，
  View/Component 可逐步收紧。
- Pytest 覆盖率报告，安全和状态机分支不能通过排除降低数字。
- 依赖漏洞审计、许可证清单、秘密扫描。
- Windows 为主平台；Python 3.10、3.11、3.12 至少覆盖核心测试。
- wheel 构建后在新虚拟环境安装，验证 `easytunnel` 命令入口。
- 发布分支额外构建桌面产物并在干净 VM 冒烟。

## 10. Windows 发布方向

### 10.1 开发入口与正式产物分离

- `start.bat` 仅用于开发，不作为正式交付入口。
- 开发入口应优先使用项目虚拟环境，避免向全局 Python/Conda 安装依赖。
- 正式产物自包含 Python/Flet 运行时，运行时不得执行 pip 或访问包索引。

### 10.2 打包选型验证

用同一组场景比较 Flet Windows build、PyInstaller/Nuitka 等可行方案：

- 包体积和冷启动。
- Defender/企业杀毒误报。
- FilePicker、托盘、Job Object、OpenSSH 调用。
- Authenticode 签名和增量升级。
- 崩溃日志和调试符号。

优先验证 `onedir`/目录型产物；单文件自解压模式可能增加冷启动和杀毒误报，应以实测决定。

### 10.3 安装包

候选为 MSIX、WiX MSI、Inno Setup 等。第一阶段至少满足：

- 默认每用户安装，不要求管理员权限。
- 正确的应用图标、版本资源、发布者、许可证和隐私说明。
- 升级前备份配置，失败可以回滚。
- 卸载时让用户选择保留或删除配置/日志；私钥从未属于应用，不得删除。
- portable ZIP 与 installer 均提供 SHA-256。

### 10.4 签名和供应链

- EXE、DLL、安装包和更新清单使用 Authenticode 和可信时间戳。
- CI 生成 SBOM、依赖清单、漏洞报告和 SHA-256 清单。
- 签名密钥不进入仓库或普通 CI 变量，使用受保护的签名服务/硬件密钥。
- 发布产物只能由 CI 生成，不从开发机手工上传。
- 未解释的高危/严重漏洞数量必须为 0。

## 11. 发布准入清单

### 功能与数据

- [ ] 所有范围内验收条件对应测试或人工用例。
- [ ] 从上一版本升级后配置、ID、设置和连接行为正确。
- [ ] 损坏配置可以恢复，失败过程不会覆盖原文件。
- [ ] 正常退出、崩溃和强制结束均无 SSH 残留。

### 安全

- [ ] 默认执行可信系统 OpenSSH，最终绝对路径可见。
- [ ] 新主机指纹确认、已知主机和密钥变化测试通过。
- [ ] 监听仍默认限制回环地址。
- [ ] 配置、日志、诊断包、剪贴板和产物通过敏感信息扫描。
- [ ] 安装包无测试私钥、生产主机、开发路径和调试数据。

### 质量

- [ ] lint、类型检查、单元、并发、集成、UI 和打包测试通过。
- [ ] 24 小时稳定性场景没有持续资源增长。
- [ ] 标准用户、Unicode/空格路径、高 DPI、无 OpenSSH 等场景有明确结果。
- [ ] Flet 固定版本与锁文件一致，依赖审计无未处理高危问题。

### 发布

- [ ] 应用、安装包、诊断信息版本一致。
- [ ] 产物签名有效且带时间戳。
- [ ] SHA-256、SBOM、变更记录和已知问题齐全。
- [ ] 干净 Windows VM 完成安装—启动—连接—升级—卸载。
- [ ] 已验证上一稳定版的回滚方案。

## 12. 推荐落地顺序

1. 可信 SSH 路径、默认示例数据清理。
2. generation/revision、刷新恢复和结构化错误码。
3. Windows Job Object、并行 shutdown、单实例。
4. 应用 known_hosts 和首次指纹确认。
5. 配置备份恢复、ACL、结构化日志和诊断包。
6. 真实 OpenSSH 集成测试和 Windows E2E。
7. 自包含打包、安装升级、SBOM、签名和发布门禁。

安全和发布工作应与功能迭代并行，不应等到 `1.0.0` 前一次性补齐。
