# 下一阶段功能规格：`0.2.0` 可靠运行

本文把路线图中的首个里程碑细化为可以直接拆 issue、写代码和验收的功能规格。建议 `0.2.0` 只包含本文件范围；命令导入、ProxyJump、反向转发等留到后续版本。

## 1. 版本完成定义

`0.2.0` 完成时，用户应能：

1. 创建或编辑一条本地 SSH 转发。
2. 从未连接状态一次点击连接并打开目标服务。
3. 明确看到“SSH 隧道状态”和“目标服务状态”。
4. 在网络短暂中断后按配置自动恢复。
5. 随时取消连接或重连，并保证不会再出现迟到的后台连接。
6. 关闭应用时选择断开退出或保留到托盘。
7. 在设置页修改 SSH 路径、默认超时、重连、日志和关闭行为。
8. 配置损坏时从备份恢复，而不是丢失原配置。
9. 导出不包含秘密的诊断包用于排障。

## 2. 功能一：自动重连

### 2.1 用户故事

> 作为需要长期使用远程桌面的用户，我希望 Wi-Fi 切换或服务器短暂重启后隧道自动恢复，而不需要反复操作开关。

### 2.2 配置模型

建议新增独立值对象，不把更多散落字段继续堆入 `TunnelConfig`：

```python
@dataclass(frozen=True, slots=True)
class ReconnectPolicy:
    enabled: bool = False
    max_attempts: int = 5          # 0 表示无限，但 UI 必须明确提示
    initial_delay_seconds: float = 2.0
    max_delay_seconds: float = 60.0
    multiplier: float = 2.0
    jitter_ratio: float = 0.2
```

延迟建议按下式计算：

```text
min(max_delay, initial_delay × multiplier^(attempt-1)) × random(1-jitter, 1+jitter)
```

### 2.3 状态与交互

新增状态：

- `WAITING_TO_RECONNECT`：显示下次尝试倒计时和当前次数。
- `RECONNECTING`：正在进行一次新的连接尝试。
- `PERMANENT_ERROR`：默认不再重试，需要用户处理。

卡片操作：

- “立即重试”：跳过等待，但不重置已经使用的次数。
- “取消重连”：关闭开关并取消待执行任务。
- “查看原因”：进入该隧道过滤后的日志页。

### 2.4 错误分类

默认可重试：

- 超时、连接拒绝、网络不可达、DNS 暂时失败。
- 已连接进程的意外退出。
- 服务器保活失败。

默认不可重试：

- `Permission denied (publickey)`。
- 私钥不存在或不可读。
- 主机密钥发生变化。
- 本地端口被其他进程占用。
- 配置校验失败。

用户可以手动重试永久错误，但程序不能自动形成认证失败风暴。

### 2.5 验收标准

- 用户主动关闭开关后，已排队的重连任务在 200 ms 内进入取消状态，之后不创建新进程。
- 第 N 代连接的迟到回调不能修改第 N+1 代连接状态。
- 达到最大次数后进入永久错误，且保留最后一次原始错误。
- 应用 shutdown 时取消全部计时器和任务。
- 测试使用可控时钟，不允许测试真实等待 60 秒。

## 3. 功能二：目标服务健康检查

### 3.1 用户故事

> 作为用户，我希望区分“SSH 已连接”和“内网的 RDP/Web 服务真的可访问”，避免绿色开关给出错误安全感。

### 3.2 状态模型

建议新建 `ServiceHealth`，不要把服务异常直接改成 SSH `ERROR`：

```python
class HealthState(str, Enum):
    UNKNOWN = "unknown"
    CHECKING = "checking"
    HEALTHY = "healthy"
    UNREACHABLE = "unreachable"
    DISABLED = "disabled"
```

卡片可以组合显示：

| 隧道 | 服务 | 界面文案 |
| --- | --- | --- |
| 已连接 | 健康 | 隧道与目标服务均可用 |
| 已连接 | 不可达 | 隧道运行中，目标服务不可达 |
| 已连接 | 未知 | 隧道运行中，尚未检查目标服务 |
| 已断开 | 任意 | 未连接，不执行探测 |

### 3.3 探测策略

- 实际探测应连接本地转发端口，而不是从本机直接连接内网地址。
- 默认在隧道进入 `CONNECTED` 后探测一次，超时建议 2 秒。
- 周期探测为高级选项，默认关闭；最短周期不低于 10 秒。
- 一次 TCP 成功只表示端口可连接，不代表应用层认证成功。
- RDP、数据库等服务可能记录连接；UI 必须提示探测行为。
- 每个隧道同时最多存在一个探测任务。

### 3.4 验收标准

- 目标服务关闭时 SSH 进程保持运行，UI 显示降级而不是关闭隧道。
- 探测超时不阻塞 Flet UI 线程。
- 断开隧道会立即取消进行中的探测。
- IPv4、IPv6 回环地址均有测试。

## 4. 功能三：一次点击“连接并打开”

### 4.1 用户故事

> 作为日常使用者，我希望点击一次卡片主按钮就完成连接并打开远程桌面或网页，而不需要先操作开关、等待、再点第二个按钮。

### 4.2 交互规则

- 未连接时主按钮为“连接并打开远程桌面/网页”。
- 点击后记录一次性 `OpenAfterConnected(tunnel_id, generation)` 意图并开始连接。
- 只有同一 generation 进入 `CONNECTED` 时才能执行一次打开动作。
- 连接中再次点击只提供取消，不能再排队一个打开动作。
- 已连接时按钮只执行打开，不重新连接。
- 自动连接和自动重连默认不执行打开动作。
- “仅连接”保留为次级菜单项，适合数据库客户端等由用户稍后使用的场景。

### 4.3 服务动作

- RDP：`打开远程桌面`。
- Web：根据配置的 `http/https`、路径和本地端点打开。
- 通用 TCP：默认复制地址；未来可选择受控客户端。
- 所有本地客户端仍通过参数列表启动，不接受任意 Shell 模板。

### 4.4 验收标准

- 从断开状态一次点击后，成功只打开一个客户端窗口。
- 快速双击、连接失败、取消、旧回调迟到均不会重复打开。
- 自动重连成功不会在用户工作期间突然弹出客户端。
- 打开客户端失败不关闭已经建立的 SSH 隧道，而是显示独立错误。

## 5. 功能四：可靠关闭与后台运行

### 5.1 用户故事

> 作为桌面用户，我希望点击关闭按钮时明确知道隧道会怎样处理，并且退出后不会留下无法识别的 ssh.exe。

### 5.2 关闭策略

新增全局设置 `close_behavior`：

- `ASK`：首次或每次询问。
- `DISCONNECT_AND_EXIT`：停止所有隧道后退出。
- `MINIMIZE_TO_TRAY`：窗口隐藏，隧道继续运行。

关闭确认对话框应展示活动隧道数量，并提供“记住我的选择”。不能提供“退出但故意让无监管子进程继续运行”的选项。

### 5.3 进程监管

Windows 推荐使用 Job Object，并启用 `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE`：

- 每个由应用创建的 SSH 进程立即加入 Job。
- 应用正常关闭、崩溃或被终止时，系统负责回收 Job 中的子进程。
- 若未来使用 `ProxyJump`/代理子进程，也应验证是否被纳入监管范围。

增加单实例机制：

- 首实例持有命名互斥量或锁文件，并监听激活请求。
- 第二实例只通知首实例显示窗口，然后退出。
- 锁文件必须包含 PID 与进程创建时间，防止旧锁误判。

### 5.4 验收标准

- 连接中关窗、重连等待中关窗、停止卡住时关窗均不遗留本次创建的 SSH 进程。
- 从托盘退出与窗口退出调用同一个 shutdown 用例。
- 系统注销/关机事件执行有时限的快速清理。
- 多实例测试不会误杀其他用户手工启动的 SSH。

## 6. 功能五：可保存的应用设置

### 6.1 数据模型

建议把应用级设置与隧道配置分开：

```python
@dataclass(slots=True)
class AppSettings:
    ssh_executable: str = ""
    default_identity_directory: str = ""
    default_connect_timeout: int = 10
    default_keepalive_interval: int = 15
    default_reconnect_policy: ReconnectPolicy = field(default_factory=ReconnectPolicy)
    close_behavior: str = "ask"
    start_with_windows: bool = False
    log_level: str = "info"
    log_retention_days: int = 7
    max_log_size_mb: int = 20
    theme_mode: str = "system"
```

推荐使用单独的 `settings.json`，使隧道导入导出不会意外覆盖应用行为。

### 6.2 设置页交互

- “SSH 可执行文件”：路径输入、浏览、自动检测、执行 `ssh -V` 测试。
- “启动行为”：开机启动、关闭行为、是否恢复上次连接。
- “连接默认值”：超时、保活、重连策略；只影响以后新建的隧道。
- “日志”：级别、保留、打开日志目录、清理日志。
- “外观”：系统/浅色/深色。

保存采用候选副本：先校验并写入临时文件，成功后才更新运行时。修改 SSH 路径不能偷偷重启当前隧道，应提示“仅对下次连接生效”。

### 6.3 开机启动

Windows 第一阶段可使用当前用户的 Startup Task 或注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`。必须：

- 只写当前用户范围，不请求管理员权限。
- 设置关闭时移除自己创建的项。
- 安装路径变化后能更新或清除失效项。
- 源码运行模式若路径不稳定，应禁用此功能并说明原因。

## 7. 功能六：配置备份与恢复

### 7.1 保存策略

在现有原子替换基础上增加：

1. 序列化并完整校验候选配置。
2. 将当前有效配置复制为带时间和版本的备份。
3. 写入同目录唯一临时文件并 `fsync`。
4. 原子替换正式文件。
5. 清理超出保留数量的最旧备份。

建议保留最近 5 份，备份同样不能包含密码或私钥内容。

### 7.2 恢复界面

读取正式文件失败时：

- 不把示例配置自动写回正式文件。
- 展示错误详情、正式文件位置和每个备份的时间/隧道数量。
- 提供“从备份恢复”“导出损坏文件”“以空配置启动”三个动作。
- 以空配置启动也不能覆盖损坏文件，直到用户明确保存。

### 7.3 schema 迁移

每个迁移函数只做一件事：

```python
MIGRATIONS = {
    1: migrate_v1_to_v2,
    2: migrate_v2_to_v3,
}
```

迁移流程必须可重复测试，并先备份旧文件。高于当前程序支持的 schema 继续拒绝加载，避免降级程序破坏新配置。

## 8. 功能七：结构化日志与诊断包

### 8.1 日志事件

日志至少包含以下字段：

```text
timestamp, level, event_code, tunnel_id, generation,
state_before, state_after, attempt, message, details
```

建议事件码稳定且与中文文案分离，例如：

- `TUNNEL_START_REQUESTED`
- `SSH_PROCESS_STARTED`
- `LOCAL_FORWARD_READY`
- `SERVICE_HEALTH_FAILED`
- `RECONNECT_SCHEDULED`
- `SSH_AUTH_FAILED`
- `TUNNEL_STOPPED_BY_USER`

### 8.2 持久化与脱敏

- 使用按大小轮转的 UTF-8 JSON Lines 或标准 logging RotatingFileHandler。
- 私钥内容、口令、完整环境变量绝不写入日志。
- 私钥路径默认只保留文件名；用户、主机和内网地址在诊断包中可选择哈希或替换。
- UI 日志只加载最近一页，不能一次读入整个历史文件。

### 8.3 诊断包

包含：

- 应用、Python、Flet、OpenSSH 和 Windows 版本。
- `ssh -V` 输出。
- 脱敏后的设置与隧道结构。
- 最近的结构化日志。
- 本地端口占用结果和应用数据目录权限摘要。

不包含：私钥、known_hosts 原文、密码、ssh-agent 中的密钥材料、完整用户环境。

## 9. 功能八：表单与连接诊断

### 9.1 表单优化

- 分为服务信息、SSH 连接和折叠的高级设置。
- 暴露当前模型已有的连接超时、保活间隔，并为 RDP、HTTP/HTTPS、PostgreSQL、MySQL 提供端口预设。
- 字段失焦时局部校验，不只在表单底部显示第一条错误。
- 端口冲突可一键选择推荐的空闲端口。
- 关闭存在未保存修改的表单时确认；保存失败保留全部输入。
- 使用响应式宽度，支持高 DPI、Tab、Enter 和 Esc。

### 9.2 分步骤测试连接

按以下步骤展示成功/失败/跳过和耗时：

1. OpenSSH 路径及版本。
2. 配置格式和私钥/ssh-agent。
3. 本地端口。
4. SSH 主机网络及认证。
5. 本地转发监听。
6. 目标服务健康。

测试使用临时 generation，结束或取消后必须清理进程和端口，不影响其他运行隧道。错误展示“原因 / 建议操作 / 原始信息”三层内容。

## 10. 建议 issue 拆分

以下拆分能让每个改动保持可评审：

1. `domain: 新状态和 TunnelEvent 模型`
2. `manager: generation token 与取消语义`
3. `manager: ReconnectPolicy 调度器`
4. `ui: 重连状态与操作`
5. `health: 单次 TCP 服务探测`
6. `ui: 双层健康状态`
7. `application: 一次性连接后打开意图`
8. `ui: 卡片连接并打开主操作`
9. `platform/windows: Job Object 监管`
10. `platform/windows: 单实例与窗口激活`
11. `settings: AppSettings 存储和迁移`
12. `ui: 可编辑设置页`
13. `storage: 配置备份与恢复服务`
14. `logging: 结构化事件和轮转`
15. `diagnostics: 分步骤测试与脱敏诊断包`
16. `ui: 响应式表单与字段级校验`
17. `release: 0.2.0 回归矩阵与迁移测试`

每个 issue 必须包含正常路径、失败路径、取消/关闭路径和自动化测试，不把“UI 做完”视为功能完成。

## 11. 本版本明确不包含

- SSH 密码保存或私钥口令输入。
- ProxyJump、反向转发、SOCKS、多转发会话。
- 配置云同步。
- 任意自定义 Shell 命令。
- 自动下载或替换系统 OpenSSH。

这些边界可以控制 `0.2.0` 规模，让团队集中解决可靠性基础。
