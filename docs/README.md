## 优先级约定

- **P0**：影响隧道正确性、进程回收、配置安全或基本可用性；进入下一版本前必须完成。
- **P1**：显著提升日常效率和可诊断性；应进入随后 1～2 个版本。
- **P2**：高级场景或体验增强；在核心架构稳定后实现。
- **P3**：生态和长期能力；按真实用户反馈决定是否投入。

工作量使用相对等级，不等同于承诺日期：

- **S**：约 0.5～2 个理想开发日。
- **M**：约 3～5 个理想开发日。
- **L**：约 1～2 个理想开发周。
- **XL**：需要拆分为多个里程碑。

## 建议的下一步

不要立即加入反向转发、SOCKS、云同步等大量新入口。建议先完成一个以“可靠运行”为目标的 `0.2.0`：

1. 应用关闭和 SSH 子进程回收可靠化。
2. 断线自动重连及退避策略。
3. 区分“SSH 隧道已建立”和“目标服务可用”。
4. 把当前只读设置页改为可保存的真实设置。
5. 建立结构化事件、持久日志和诊断导出。
6. 将 1000 行以上的 `app.py` 逐步拆成视图、组件和控制器。

完成上述基础后，再实现命令导入、系统托盘、多跳连接和其他转发模式，返工成本会明显更低。

## 文档维护规则

- 每个版本开始前，从路线图中选定范围，并把验收条件转为 issue 或测试用例。
- 配置字段、状态机、命令参数变化时，同步更新架构文档。
- 安全默认值发生变化时，必须记录原因、迁移行为和回滚方式。
- 已实现功能从“计划”移入 README/CHANGELOG，不让路线图长期混合完成项与待办项。
- 版本发布后记录实际投入和遗漏，用于校准后续工作量等级。
